# План: полнота архива (страницы + аудио + ссылки) — 2026-07-16

Основание: `docs/audit/pages-inventory-2026-07-16.md` (полная картина слепок vs БД).
Черновые данные прогонов: скретчпад сессии (`missing_pages.json`, `missing_site_real.json`,
`missing_audio.json`, `mp3_inventory.json`, `link_report.json`) — при реализации пересчитать заново кодом команд.

**Корневая причина недостачи:** все импортёры и аудит перечисляют только корень слепка
(`File::glob($base.'/index.php@title=*')` в `ImportOfflineWiki`/`ImportOfflineAudio`/`AuditArchive`,
`glob('*/default.htm')` глубины 1 в `ImportOfflineExplorer`). Offline Explorer разложил часть страниц
по подпапкам `%&Ovr0`…`%&Ovr119` — они были невидимы. Пример: Сеанс с силами 20070730b лежит в
`%&Ovr3/`, его mp3 — в `%&Ovr66/20070730b.mp3`.

## Инварианты (действуют на КАЖДОМ этапе)

1. **Опубликованное не трогаем.** Ни одна команда не пишет в `pages` со `status='published'`,
   кроме двух явно разрешённых точечных операций: (а) вставка отсутствующей ссылки вокруг
   существующего якорного текста, (б) удаление табличной строки «Аудио Запись». Обе — через
   ревизию с внятной пометкой («Восстановлена ссылка …» / «Удалена строка Аудио Запись»),
   чтобы история была видна и защита `--refresh` не ломалась (пометка НЕ «Отредактирована вручную»).
2. **Общий гард в коде, не в головах.** Завести хелпер/трейт (например, `GuardsPublished`)
   с единственным методом изменения страницы; все команды этого плана пишут только через него.
   Перед любым прогоном — файловый бэкап `database.sqlite` с датой (сделать это частью команд:
   первая пишущая операция за прогон создаёт копию).
3. **Идемпотентность.** Повторный запуск любой команды не создаёт дублей и не меняет уже
   исправленное — как у `site:content-fixes-2026`.
4. **Не восстанавливать удалённое пользователем.** Константа-исключение (титулы в нижнем регистре):
   «История создания», «Правила X-INTELLECT», «Политика конфиденциальности»,
   «Политика использования Cookies», «Александр Георгиевич Глаз - основатель проекта»,
   «Архивы курсов Александра Глаза». Пополняется вручную.
5. **Авторство git:** `Fesoterika <linewery@gmail.com>`, без упоминания ботов.

## Этап 1. Общий индекс слепка (фундамент, делает всё остальное согласованным)

Новый сервис `App\Services\OfflineSnapshotIndex` (или метод в `MediaWikiArchive`):

- обходит `wiki/` корень **и все** `%&OvrN`;
- берёт файлы `index.php@title=*` без `&` (канонические, не diff/printable/edit);
- заголовок и namespace — из `mw.config` в HTML (`"wgTitle"`, `"wgNamespaceNumber":0`,
  `"wgAction":"view"`), НЕ из имени файла (имена обрезаны и двойно закодированы);
- тело — `<div id="mw-content-text">…<div class="printfooter">`;
- отбрасывает заглушки: «В настоящее время на этой странице нет текста»,
  «текст на данной странице отсутствует», редиректы, тела короче 25 знаков;
- при нескольких вариантах одной страницы выбирает самое длинное тело;
- отдаёт: title, путь к лучшему файлу, все варианты (для союза mp3-ссылок).

Переключить на этот индекс: `ImportOfflineWiki`, `ImportOfflineAudio`, `AuditArchive`.
Тест: индекс находит «Сеанс с силами 20070730b» и суммарно ~272 статьи (без падения старых цифр).

## Этап 4. Аудио

`import:offline-audio` расширить (или новая `audio:backfill`):

- источник ссылок — новый индекс (этап 1), союз mp3-ссылок по всем вариантам страницы;
- **сопоставление по дате**: если у страницы с датой в заголовке (`(19|20)\d{6}[a-z]?`)
  нет аудио — искать mp3 по этой дате в папках, приоритет:
  1) слепок Offline2015 (включая `%&OvrN/*.mp3`),
  2) `/Volumes/KINGSTON/…/Сайт и файлы + аудио`,
  3) `/Volumes/KINGSTON/…/Архив Дмитрия Морозова` (в последнюю очередь);
- при совпадении размера файла в нескольких папках — брать первый по приоритету;
  разные размеры = разные дорожки (например 20110508.mp3.mp3 и 20110508k.mp3 — обе);
- копирование в `storage/app/public/media/audio/`, запись в `media` (type=audio,
  page_id, position по порядку), названия дорожек по правилам `trackTitle`;
- привязка аудио к странице НЕ меняет `pages.body` → безопасно и для published
  (media-связь, не тело); но для аккуратности логировать и published-привязки в отчёт;
- уже найденные 11 страниц без аудио (см. отчёт) — первые кандидаты; плюс все
  доимпортированные сеансы (у 20070730b в слепке есть и mp3, и ссылки).

## Этап 6. Восстановление ссылок (прогон по всем материалам)

Новая команда `links:restore {slepok} [--dry] [--published-only]`:

1. Для каждой страницы с архивным исходником — набор ссылок исходника, фильтруя хром
   MediaWiki: «Версия …», «Текущая версия …», «править», «просмотреть исходный код»,
   «← Предыдущая правка», «Следующая правка →», «thumb», ссылки на Участник:/Служебная:,
   красные (`class="new"`), самоссылку-заголовок, чисто числовые якоря.
2. Маппинг цели: старый wiki-title → новая цель (pages по title → glossary_terms →
   таблица redirects → иначе «цель отсутствует»). Если цель входит в списки исключений —
   страницы вне охвата (этап 3), удалённые пользователем (инвариант 4) — ссылку
   НЕ вставлять, в отчёт со статусом «вне охвата». По прогону 16.07: к восстановлению
   108 ссылок (из них 44 в published), вне охвата — 6.
3. Если якорный текст присутствует в теле БД и не находится внутри `<a>` — **точечная**
   вставка `<a href>` вокруг первого вхождения (DOM, без переписывания остального тела),
   сохранение через гард с ревизией. Работает и на published — это разрешённая операция.
4. Если текст исходника в теле не найден (правлено вручную) — только в отчёт, не трогать.
5. `--dry` по умолчанию выводит полный список планируемых вставок; боевой прогон — после
   просмотра. Первый прогон — по published (5 страниц из отчёта: Дайджест сентябрь 2012
   «ссылке», Курсы (9 ссылок на планы тренингов — появятся после этапа 2), Чёрные
   («Зеленые», «Голубые»), Картины Учителей Ноосферы («Сеанс с силами 20070730b»),
   Развитие энергоинформационного восприятия).

Порядок важен: этап 6 — ПОСЛЕ этапов 2–3 (цели ссылок должны существовать).
